# LLM 职责划分架构原则

## 核心思想

> **LLM 做擅长的事，复杂计算等不擅长的，交给工具/函数/程序/模型（稳定无 Bug）**
> 
> — 用户反馈

---

## 1. 职责边界定义

### LLM 擅长做的事 ✅

| 能力 | 具体表现 | 示例 |
|:---|:---|:---|
| **意图理解** | 从自然语言提取用户需求 | "创建登录表单" → 识别为 FORM |
| **语义推理** | 推断应包含哪些子组件 | FORM → [用户名, 密码, 登录按钮] |
| **内容生成** | 生成有意义的文本内容 | 按钮文案、标题、描述 |
| **上下文关联** | 根据对话历史调整输出 | "改成深色主题" |
| **模糊决策** | 在多个合理选项中选择 | 选择合适的颜色搭配 |

### LLM 不擅长做的事 ❌ → 交给工具

| 能力 | 具体表现 | 应该交给 |
|:---|:---|:---|
| **精确数值计算** | padding: 16 还是 24? | `tokenSlot.resolve()` |
| **约束传播** | HUG 父节点不能有 FILL 子节点 | [resolveLayoutMode()](file:///Users/daxiaoxiao/Projects/figma%20gen%20plugin/figma-ai-generator/src/services/rendering/layerRenderer.ts#614-643) |
| **一致性保证** | 同样的 CARD 每次 padding 相同 | `ANATOMY_REGISTRY` |
| **格式转换** | semantic → Figma API 属性 | [sanitizeLayer()](file:///Users/daxiaoxiao/Projects/figma%20gen%20plugin/figma-ai-generator/src/services/rendering/layerRenderer.ts#9-239) |
| **枚举验证** | layoutMode 只能是 H/V/N | JSON Schema |

## 2. 过度工程化评估框架

### 警告信号 ⚠️

| 信号 | 描述 | 例子 |
|:---|:---|:---|
| **为 LLM 优化格式** | 牺牲可读性换取 Token 节省 | `l:H;p:16` 替代 `layoutMode:HORIZONTAL` |
| **复杂转换层** | 需要双向转换器 | `compactProps() ↔ expandProps()` |
| **隐式约定** | 位置含义替代属性名 | `[0,1,2,3]` 代表 `[T,R,B,L]` |
| **过早优化** | 在验证有效性前优化性能 | 还没确认语义模板有效就设计转换层 |

### 评估矩阵

| 方案 | 复杂度 | 可维护性 | Token 节省 | 推荐度 |
|:---|:---|:---|:---|:---|
| 当前 JSON (不变) | ⭐ | ⭐⭐⭐ | 0% | ✅ 基线 |
| 语义模板 (frameType) | ⭐⭐ | ⭐⭐⭐ | ~50% | ✅ 推荐 |
| 缩写枚举 (H/V/N) | ⭐⭐ | ⭐⭐ | ~20% | ⚠️ 视需求 |
| 紧凑数组格式 | ⭐⭐⭐ | ⭐ | ~60% | ❌ 过度工程化 |

## 3. 推荐架构：语义驱动 + 确定性工具

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                            用户输入                                         │
│  "创建一个深色主题的登录卡片"                                               │
└────────────────────────────────┬────────────────────────────────────────────┘
                                 ↓
┌─────────────────────────────────────────────────────────────────────────────┐
│                     LLM 输出 (语义层)                                       │
│  只输出 LLM 擅长的：                                                        │
│  {                                                                          │
│    semantic: "CARD",                                                        │
│    variant: "dark",                                                         │
│    title: "登录",                                                           │
│    children: [                                                              │
│      { semantic: "TEXT_FIELD", label: "用户名" },                           │
│      { semantic: "TEXT_FIELD", label: "密码", secure: true },               │
│      { semantic: "BUTTON", label: "登录" }                                  │
│    ]                                                                        │
│  }                                                                          │
└────────────────────────────────┬────────────────────────────────────────────┘
                                 ↓
┌─────────────────────────────────────────────────────────────────────────────┐
│                   工具层 (确定性计算)                                       │
│                                                                             │
│  1. tokenSlot.resolve('CARD', 'dark')                                       │
│     → { padding: 24, cornerRadius: 12, fills: ['$card-dark'] }              │
│                                                                             │
│  2. ANATOMY_REGISTRY['card']                                                │
│     → { layoutMode: 'VERTICAL', gap: 16 }                                   │
│                                                                             │
│  3. resolveLayoutMode(sizing, nodeType, parentLayout)                       │
│     → 修正无效的 HUG/FILL                                                   │
│                                                                             │
│  4. sanitizeLayer(layer)                                                    │
│     → 注入缺失的子结构                                                      │
└────────────────────────────────┬────────────────────────────────────────────┘
                                 ↓
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Figma JSON (渲染就绪)                                │
│  {                                                                          │
│    type: "FRAME",                                                           │
│    props: {                                                                 │
│      name: "登录卡片",                                                      │
│      layoutMode: "VERTICAL",     // ← 工具推导                              │
│      padding: 24,                // ← tokenSlot                             │
│      cornerRadius: 12,           // ← tokenSlot                             │
│      fills: ["$card-dark"],      // ← tokenSlot                             │
│      gap: 16                     // ← ANATOMY_REGISTRY                      │
│    },                                                                       │
│    children: [...]               // ← 完整结构                              │
│  }                                                                          │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 4. 用户掌控感设计

### 原则：用户可以面向 LLM 获得掌控感

| 场景 | 用户说 | LLM 应该理解 | 工具应该做 |
|:---|:---|:---|:---|
| **覆盖默认值** | "padding 用 32" | `padding: 32` (用户指定) | 使用 32，不用 tokenSlot |
| **解释约束** | "为什么不能 HUG?" | 返回解释 | 提供修正日志 |
| **迭代调整** | "改成横向排列" | `layoutMode: HORIZONTAL` | 不覆盖用户意图 |

### 三级意图优先级

```
优先级 1: 用户明确指定 (不修正)
  "padding 必须是 8"
  → 后处理层不覆盖，即使与 tokenSlot 不一致

优先级 2: LLM 推理 (谨慎修正 + 反馈)
  LLM 输出 HUG，但 parent 是 NONE
  → 修正为 FIXED，并反馈原因

优先级 3: 工具默认值 (随时可覆盖)
  LLM 未指定 padding
  → 使用 tokenSlot 默认值 16
```

## 5. 假设总结

| 假设 | 内容 | 置信度 |
|:---|:---|:---|
| **H-Z** | 最小工程化：LLM 只做语义，工具做计算 | 95% |
| **H-X** | 语义模板：用 semantic 替代布局属性 | 90% |
| **H-U** | 三级意图优先级：用户 > LLM > 工具默认 | 85% |
| **H-T** | 修正日志反馈：后处理修正需要告知 LLM | 90% |

## 7. 自我批评

### 优点
- 整合了用户的软件工程视角
- 定义了清晰的职责边界
- 提供了过度工程化评估框架

### 改进方向
- 需要实际验证"语义模板"模式的生成质量
- 需要量化"用户掌控感"的体验提升
