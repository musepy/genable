## Vision

### Core Thesis

> "Figma 生态内唯一一个住在画布里的 AI agent。"

不是外部生成器（Galileo/Uizard），不是 copilot 弹窗（Figma AI）。
Agent 住在你的文件里，读你的设计系统，记住你的偏好，注意到你的变化。

### 10x Check

10x 版本 = **"设计师说：agent 比我更了解我们的设计系统"**

- 打开任何文件 → agent 3 秒内知道你的色板、字体、间距规范
- 跨文件工作 → agent 自动迁移设计系统知识
- 设计系统变更 → agent 主动说"我注意到你改了主色，要我更新所有相关页面吗？"
- 新设计师入职 → 打开 .agent page 就是完整的设计系统文档

当前 Approach B 能覆盖前 2 点（warm start + memory），为后 2 点留出结构。

### Product Identity — Three Sentences

1. **What it IS**: A canvas-native AI agent that lives inside your Figma file, understands your design system, and remembers your preferences across sessions.
2. **What it ISN'T**: Another text-to-UI generator that dumps designs and forgets you.
3. **Why it matters**: Design work is contextual — your brand, your system, your decisions. An agent without memory is just a fancy template engine.

## Resolved Decisions (was Open Questions)

### 1. 记忆页命名：`.agent`
**决定**: 用 `.agent`（dot-prefix 表示系统级，类比 `.git`、`.env`）。
**理由**: 已有 agent-page skill 全部使用 `.agent` 命名；vision doc 也用这个名字；用户理解"以 dot 开头的是系统文件"。

### 2. Memory Diff 的快照存储：pluginData
**决定**: 用 `node.setPluginData('lastSnapshot', JSON.stringify(tokenSnapshot))` 存在 .agent page 根节点上。
**理由**: pluginData 是 per-plugin 隔离的（不会和其他插件冲突），自动跟 Figma 文件保存，不占 clientStorage 配额。快照内容是 design tokens（colors/fonts/spacing），不是 node-level 状态——token 稳定性远高于节点 ID。

### 3. 超链接语法：`[node:ID]`
**决定**: `[node:1234:5678]`（Figma node ID 格式）。MessageRenderer 解析后渲染为可点击标签，点击执行 `figma.viewport.scrollAndZoomIntoView([node])`。
**理由**: 比 `[link:NODE:xxx]` 更简洁，node ID 可从工具结果直接获取。

## Memory Architecture

### 决策：保留 clientStorage（Eng Review 2026-03-21 修正）

**Eng Review 修正**：CEO plan 原计划迁移到 pluginData，但工程审查后决定保留 clientStorage。

**理由**：
1. pluginData 需要 .agent page 先存在——onboarding 时鸡生蛋问题
2. 多人协作共享记忆是 Approach C 范围，当前 B 不需要
3. clientStorage 已能工作，用户体验完全相同（agent 行为不变）
4. 迁移增加的复杂度无当前用户价值（Brooks: essential vs accidental complexity）

**存储布局**：
| 存储 | 内容 | Key 格式 |
|------|------|---------|
| `figma.clientStorage` | Memory entries | `mem:<key>` (已有) |
| `figma.clientStorage` | Token snapshot (for diff) | `_token_snapshot` (新增，独立于 memory CRUD) |
| `.agent/memory/` text nodes | 人类可读摘要（展示层） | agent 更新，用户只读 |

**未来迁移 (TODO P2)**：当多人协作需求确认后，迁移到 pluginData on .agent page root node。

### Memory Format

```typescript
interface MemoryEntry {
  key: string           // e.g., "brand-colors", "typography", "user-preferences"
  value: string         // 自然语言描述 + 结构化数据混合
  updatedAt: string     // ISO 8601
  source: 'agent' | 'user'  // 谁写的
}

// 示例
{
  key: "brand-colors",
  value: "Primary: #2563EB (Blue 600), Secondary: #475569 (Slate 600). User prefers high contrast ratios.",
  updatedAt: "2026-03-21T10:30:00Z",
  source: "agent"
}
```

**Size budget**: 单条 ≤2KB，总量 ≤20KB（约 10 条记忆）。超过时，agent 合并旧条目或提示用户清理。

### Token Snapshot Format (for Memory Diff)

```typescript
interface TokenSnapshot {
  colors: Record<string, Record<string, string>>  // scale/step → mode → hex
  fonts: Record<string, string>     // role → "family/size/weight"
  spacing: Record<string, number>   // name → px value
  timestamp: string
}
```

**V1 scope**: Value-level diff only（检测 hex 值变化、新增/删除 key）。变量重命名检测和多 mode diffing 是 V2。

**Concurrency guard**: Token snapshot 写入前检查 `timestamp`——如果比读取时更新，说明其他用户已写入，跳过本次写入（optimistic lock），避免 phantom diffs。

### Onboarding Scan Algorithm

扫描不依赖 page 命名启发式——直接用 Figma API 获取结构化数据：

1. `figma.variables.getLocalVariablesAsync()` → 获取所有本地变量（颜色、数字）
2. `figma.getLocalTextStylesAsync()` → 获取字体样式
3. `figma.getLocalPaintStylesAsync()` → 获取颜色样式

从这三个 API 的结果中提取 design tokens。Success criteria "≥80% 正确率"的 ground truth = 文件实际的 local styles/variables 数量。

**Scan limits**: 这三个 API 调用是全局的（不按 page），所以 100+ page 的文件也只需 3 次 API 调用。如果变量超过 500 个，只取最常用的前 50 个（按 usage count 排序）。

## Edge Cases & Degradation

| 场景 | 行为 |
|------|------|
| .agent page 被用户删除 | 下次 run() 检测到不存在 → 重新走 onboarding 流程 |
| Memory 数据损坏/无法解析 | 静默跳过，log warning，不阻塞 agent 启动 |
| Memory 超过 20KB | Agent 提示用户清理 or 自动合并最旧的 3 条 |
| 空文件（无设计系统） | Onboarding 扫描发现无内容 → 跳过初始化，等用户首次设计后再写 |
| 大型企业文件（100+ 页） | Onboarding 扫描限制：只扫当前 page + 名称含 "design system" / "style" 的 page，最多 50 个顶层 frame |
| **多人协作**（Figma multiplayer） | Memory 存在 pluginData 中（per-plugin 但 shared across users）。采用 last-write-wins 策略——设计系统级别的记忆本质上是文件共享的事实，不存在"我的记忆 vs 你的记忆"。如果两个 agent 同时写入，后写的覆盖先写的是可接受的，因为内容应该一致（都是从同一个画布读取的） |
| read-only 文件权限 | 检测写入失败 → fallback 到只读模式（读已有记忆但不更新）。不弹错误 |
| Token snapshot 中 node ID 失效 | Diff 基于 token 值（色值/字体名），不基于 node ID。node 重建不影响 diff |

## Success Criteria

| 里程碑 | 可测量指标 |
|--------|-----------|
| Canvas Memory 读写 | Warm start 注入记忆 < 500ms；turn 结束写入 < 200ms |
| Onboarding 扫描 | 扫描 < 5s（50 page 以内）；正确识别 ≥80% 的色板/字体 page |
| Memory Diff | Token-level diff 正确率 ≥90%（颜色变更、字体变更） |
| Prompt Chips | 用户点击 chip 后 agent 正确执行记忆操作 |
| Design System 超链接 | 点击链接后 viewport 跳转到正确节点 |
| 整体 | Demo 场景（3 分钟）端到端无阻塞完成 |

## Prompt Directives (记忆管理指令)

添加到 WORKFLOW.md 的具体内容：

```markdown
## MEMORY MANAGEMENT

On each turn end, evaluate: did the user establish any REUSABLE design decisions?

WRITE memory when:
- User specifies brand colors, fonts, or spacing preferences
- User says "always" / "from now on" / "remember" about a design choice
- A design system pattern is established (e.g., "all cards use 12px corner radius")

DO NOT write memory when:
- One-off styling choices ("make this button red")
- Temporary experiments
- Layout decisions that only apply to this specific design

Memory format: natural language + structured data. Example:
  Key: "typography"
  Value: "Headlines: Space Grotesk 32px Medium. Body: Inter 16px Regular. User prefers generous line height (1.5+)."

On warm start: read memory → briefly acknowledge what you remember → proceed.
  GOOD: "I see your brand uses #2563EB with Inter. I'll keep it consistent."
  BAD: "Loading memory... Found 5 entries... Entry 1: brand-colors..."
```

## UI Design Decisions (Design Review 2026-03-21)

### Warm Start 记忆确认
**方式**: 融入 agent 第一条回复，不额外 UI 元素。
- GOOD: "I see your brand uses #2563EB with Inter. I'll keep it consistent."
- BAD: 独立的 "Memory loaded" 卡片
- **原则**: "不会被注意到 = 完美"。记忆像好朋友自然记得，不是系统报告。

### Memory Badge (空状态)
- **位置**: Sparkles 图标下方，同一行
- **样式**: `textSecondary` 色 (gray-11)，11px，如 "3 items remembered"
- **条件**: 只在 memoryCount > 0 时显示

### Prompt Chips (记忆快捷动作)
- **条件显示**: 有记忆时显示 memory chips，无记忆时只显示默认 design chips
- **3 个 memory chips**:
  - 🎨 "Show design system" → 触发 `cat /.agent/memory/`
  - 🧠 "Update memory" → 引导用户输入新偏好
  - 🗑️ "Forget preference" → 选择后 `rm /.agent/memory/key`

### Onboarding 引导
- **方式**: Agent 在对话中问 "I found your design system. Want me to set up memory?"
- **确认**: 用户在对话中回复确认，不需要额外 UI 按钮
- **空文件**: "No design system found. Start designing and I'll learn as we go."

### Node Links [node:ID] (Phase 2)
- **样式**: 内联芯片标签（accent background, 4px 圆角, hover 态加深）
- **失效处理**: node 不存在时显示灰色标签 + "Node not found" tooltip
