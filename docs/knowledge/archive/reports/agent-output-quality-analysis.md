# Agent Output Quality Analysis

> Analyzed from: Figma design `580:2598` (Login Card) in `genable_dev`
> Date: 2025-03-01

## Context

A Login Card generated by the agent exhibited severe layout issues: all text displayed vertically (1-2 chars per line), components stacked into a 100×1343px strip, and divider lines rendered in wrong orientation. This report identifies the root causes, observable patterns, and actionable improvements across five dimensions.

---

## 1. Quantifiable Metrics

Indicators that can be automated to detect broken agent output:

| Metric | Detection Method | Threshold |
|--------|-----------------|-----------|
| **Magic Number Rate** | Count nodes where width=100 or height=100 | > 20% of nodes is anomalous |
| **Dimension Uniformity** | Nodes sharing identical dimensions unintentionally | > 3 non-repeated nodes sharing same size |
| **Aspect Ratio Sanity** | Root container width:height ratio | Card > 1:3 is anomalous (this case: 1:13) |
| **Overflow Clip Rate** | Frames with `clipsContent=true` / total frames | > 50% warrants a warning |
| **Content Visibility** | Avg chars per text line given container width | < 5 chars/line = layout is broken |
| **Icon Size Ratio** | Icon dimension / adjacent text fontSize | < 1.0 = icons too small (this case: 12/14 = 0.86) |
| **Padding-to-Width Ratio** | Total horizontal padding / container width | > 50% = layout mismatch (this case: 80/100 = 80%) |
| **Divider Aspect** | Long edge / short edge of separator lines | < 10:1 = likely wrong orientation |

## 2. Agent Chain Tracing

### Where does `100` come from?

**Executor fallback** (`src/engine/actions/executor.ts:354-357`):
```typescript
const defaultWidth = actionType === 'createText' ? 0 : 100;
const defaultHeight = actionType === 'createText' ? 0 : 100;
```
This is in `centerNodeInViewport()` — only used for positioning estimation, **not** for setting node size.

**The real source: Figma API defaults.** When a frame is created without explicit `width`/`height`, the Figma plugin API defaults to **100×100 pixels**.

**Root cause chain:**
```
LLM output → missing width/height properties → Figma API applies 100×100 default
```

### Where does overflow clip come from?

**Not from our code.** No `clipsContent` default is set anywhere in the codebase. Figma frames default to `clipsContent = true`.

### Source attribution

| Problem | Source Layer | Root Cause |
|---------|-------------|------------|
| Dimensions all 100px | Figma API default | LLM did not output width/height |
| overflow clip on all frames | Figma API default | LLM did not output `clipsContent: false` |
| Icons 12px | LLM output | LLM reasoning error |
| Divider orientation wrong | LLM output | LLM confused axis direction (height=100 on a horizontal line) |

## 3. LLM Reasoning Gap Analysis

The prompt system **already provides** adequate sizing guidance:

- **DESIGN_AESTHETICS**: Card width 360px, padding 16px, cornerRadius 12px
- **Tool Examples**: Complete Card creation example with `width: 360`
- **SCENE_GRAPH_MODEL**: FILL/HUG/FIXED semantics explained
- **figma-layout-rules**: Root node → FIXED + `width: 400`

Yet the LLM still omitted sizing. Hypotheses:

| Hypothesis | Likelihood | Verification |
|-----------|-----------|-------------|
| Token budget truncated DESIGN_AESTHETICS section | **High** | Dump system prompt at that iteration; check if aesthetics section present |
| Tool schema marks width/height as optional | **High** | LLM skips optional fields it considers non-essential |
| LLM assumes framework provides sane defaults | Medium | Common pattern in web dev (CSS has better defaults than Figma) |
| Examples mention width too infrequently | Medium | Count how many few-shot examples include explicit dimensions |
| DESIGN_AESTHETICS only injected in EXECUTION phase | **High** | If first iteration was PLANNING, sizing knowledge was absent |

**Most critical to verify**: Whether DESIGN_AESTHETICS and EXAMPLES were present in the system prompt during the iteration that created the Login Card.

## 4. Prompt Design Improvements

### 4.1 Defensive prompt addition (low cost, high impact)

Add to SCENE_GRAPH_MODEL or tool descriptions:
```
CRITICAL: Every FRAME node MUST have explicit width and height.
Figma defaults frames to 100×100px — this is almost never correct.
Omitting width/height is the #1 cause of broken layouts.
```

### 4.2 Tool schema enforcement

Mark `width` and `height` as required for root frame creation, or add post-validation:
- If root node width=100 && height=100 → flag as likely default, trigger re-generation

### 4.3 Validator enhancement

`postOpValidator.ts` already checks ZERO_DIM. Extend with:
```
MAGIC_SIZE: width === 100 && height === 100 && isRootNode
→ "Node uses Figma default dimensions. Set explicit width/height."
```

Also add:
```
PADDING_OVERFLOW: totalPadding > 0.5 * containerWidth
→ "Padding exceeds 50% of container width — likely dimension mismatch."
```

### 4.4 Example reinforcement

Ensure multiple tool examples consistently show explicit sizing:
- Login Card: width 400-480
- Modal: width 500-600
- Input field: height 40-48
- Button: height 44-48
- Icon: width/height 20-24
