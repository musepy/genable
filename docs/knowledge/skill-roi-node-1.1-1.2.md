## Hypothesis

Two complementary edits target the create-page over-fire problem:

## Delta breakdown — what changed for the agent

**Before fix** (atomic-component prompts):
```
[user] "Design a notification settings card with toggles…"
[agent] skill({name: "create-page"})  →  loads 2.5K tokens of page workflow
        style({name: "arctic-minimal"})  →  loads 1K tokens of style
        jsx(…)  →  builds the card (correctly, but with extra token cost)
```

**After fix**:
```
[user] "Design a notification settings card with toggles…"
[agent] jsx(…)  →  builds the card directly
        get_screenshot()  →  verify
```

Per-prompt savings: ~3.5K input tokens + 1-2 wasted turns. Across many sessions this compounds.

## Side-effect audit

- **PAGE_CLEAR / DASHBOARD_CN** (legitimate page prompts): still load create-page correctly. Tool counts stable (5-11 calls vs baseline 5-7). No regression.
- **COMPONENT_SET errors in r1 (47 tools, 8 errors)**: pre-existing component-set workflow issue with `add_component_prop` Figma API ("Can only set component property definitions on a product component" + sync `findOne` in dynamic-page mode). Skill loaded correctly; errors are downstream of skill selection. **Independent of this fix → Node 1.5 candidate.**
- **DS_REQUEST errors in r2 (34 STALE_VARIABLE_FINGERPRINT)**: test environment artifact — prior runs left Blue/Gray/Red variables in the Figma file. Plugin correctly refused to overwrite. Not edit-caused.

## Verdict

**SHIP**. Strong, stable, two-round-confirmed fix. Zero regressions on legitimate skill targets.
