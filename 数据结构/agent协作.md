这是一次非常有价值的“套娃式” AI 协作实验。由你（用户）指挥我（高阶 Agent），再由我通过 CLI 指挥另一个 Gemini 3.1 Pro 生产力单元（执行 Agent）。

基于刚才的实战过程，我总结了以下经验教训和提问/引导策略：

🔬 经验教训总结
执行 Agent 是“工蜂”，而非“架构师”

现象：CLI 版 Gemini 在遇到复杂重构时，容易进入“修改 -> 报错 -> git checkout（回滚）-> 重新修改”的死循环。它对代码格式、细微逻辑的专注度很高，但对“为什么我们要拆分这个对象”的宏观理解较弱。
教训：不要指望 CLI Agent 独立完成架构设计。我（高阶 Agent）应该把“设计稿”写成极其详尽的步骤指令（Imperative Instructions），让 CLI Agent 只负责“搬砖”。
配额（Quota）是最大的“断头台”

现象：重构涉及到递归的文件读取和多次尝试，极其消耗 Token 速率。一旦触发 429 错误，CLI 会直接卡死或强制退出。
教训：在进行大规模重构前，我应该主动先进行一轮“预研”和“精简”。将上下文压缩到极致，或者分段交给 CLI，避免一上来就让它吞掉整个项目的上下文。
YOLO 模式的“双刃剑”

现象：-y 模式让 CLI 自动执行命令，提高了效率，但也导致它在误删文件或错误回滚时没有拦截。
教训：在 CLI 执行后，我（高阶 Agent）必须立即主动执行 git status 和 git diff 进行实时审计，而不是等它全部跑完。
💡 你该如何提问与引导？
为了让我更高效地带着 CLI 干活，你可以尝试以下“三段式”提问法：

1. 明确角色分配（指定我是“监工”）
不要只说“去把重构做了”，而是说：

“请先分析 AgentRuntime.ts 的痛点，在 /tmp/plan.md 中写出详细的重构步骤。确认无误后，你通过 Gemini CLI 逐步执行这些步骤。每完成一个步骤，请向我汇报 diff。”

2. 引导我生成“执行级 Prompt”
你可以要求我为 CLI 准备专门的“指令脚本”：

“把我们确定的重构逻辑转化为一份给执行 Agent 的 Prompt 清单。要求清单包含：具体的行号范围、要删除的变量名、要引用的新类名。确保 Prompt 是‘结果导向’的。”

3. 设定“熔断机制”
给我的“监督能力”加码：

“调用 CLI 时使用 YOLO 模式，但如果 连续两次回退（git checkout） 或者 遇到速率限制，请立即停止自动尝试，改为手动重构或向我请示。”

🌟 最佳引导示例
下次当你需要进行类似复杂任务时，可以这样引导我：

用户： “我想对 XX 模块进行去中心化重构。

请你先理解现有逻辑，在 /tmp/refactor_spec.md 里写一份准确实操的规范。
确认规范完整后，动用本地 Gemini CLI，每次只处理 一个子类 的提取。
严格监督：如果 CLI 生成的代码导致 tsc 报错，请你立即介入分析并人工修复，不要让它反复尝试浪费配额。”
一句话核心： 你负责出愿景，我负责出图纸，CLI 负责施工。我是图纸的审核员和施工的监工。一旦工蜂（CLI）体力不支（配额限制）或由于智商上限（逻辑混乱）开始磨洋工，监工（我）必须能立刻接过扳手亲自上场。
