# 转型路线执行方法论 (Turn to Agent)

本项目正处于从「线性渲染管线」向「Agentic 循环」转型的关键阶段。为了确保转型的稳健性与可维护性，所有参与开发的 Agent 必须严格遵循以下执行方法。

## 1. 核心资源索引

在开始任何代码编辑前，必须深入理解以下背景与计划：

- **[执行计划](./agent-transformation-guide.md)**：转型的核心蓝图。包含 4 个阶段的详细路线图，对任务优先级有决定性指导。
- **[思路 (Thinking)](./thinking.md)**：架构设计的底层逻辑，涵盖原子工具 (Atomic Tools)、MCP 协议、上下文压缩等核心概念。
- **[基础设施 (Init)](./init.sh)**：环境检查与一致性保障脚本。在新会话或环境变更后，优先运行 `./turn to agent/init.sh`。

## 2. 状态跟踪机制 (Structured Progress)

增量工作以高度结构化的方式进行跟踪：

- **[进度日志 (Markdown)](./state_tracking.md)**：供人类阅读的进度记录，记录会话增量、决策逻辑及下一步行动。
- **[状态源 (JSON)](./state_tracking.json)**：机器可读的项目快照。包含 Phase/Task 的完成百分比及健康度指标。
- **[测试跟踪 (Tests JSON)](./tests.json)**：所有功能必须先有测试 ID。通过该文件跟踪测试的通过情况，确保无回归故障。

## 3. 开发准则：TDD 与 通用逻辑

- **测试驱动开发 (TDD)**：在修改代码前，先在 `tests.json` 定义测试并编写失败的测试用例。禁止无测试的代码提交。使用可用的标准工具编写高质量、通用的解决方案。不要创建辅助脚本或解决方法来更有效地完成任务。实现一个对所有有效输入都正确工作的解决方案，而不仅仅是测试用例。不要硬编码值或创建仅适用于特定测试输入的解决方案。相反，实现实际解决问题的逻辑。

专注于理解问题需求并实现正确的算法。测试用于验证正确性，而不是定义解决方案。提供遵循最佳实践和软件设计原则的原则性实现。

- **拒绝硬编码**：实现真正的算法逻辑，而非针对特定测试用例的补丁。
- **稳健性优先**：如果发现任务不合理或测试用例有误，应立即反馈并停止执行，而非强行「修复」表象。
- **读后而写**：始终在提出代码编辑之前阅读和理解相关文件。不要推测未检查的代码。如果用户引用特定文件/路径，必须在解释或提出修复之前打开并检查它。在搜索代码以获取关键事实时要严格和坚持。在实现新功能或抽象之前，彻底审查代码库的风格、约定和抽象。

## 4. 故障处理与反馈规范 (Bug Protocol)

遇到非预期问题时，遵循 `[/gemini]` 协议：

1.  **寻求背景**：如果不确定预期行为，停止修复并向用户确认「初衷是什么？」「观察到了什么？」。
2.  **根因分析**：在修复前，必须解释你认为哪里出错了以及为什么，拒绝「创口贴式」修复。
3.  **增量验证**：添加必要的日志，编写可复现的失败测试，修复后确保测试转绿。

## 5. 长任务与上下文管理

本转型任务跨度极长，为保证执行效率：

- **深度规划**：建议投入足够的上下文窗口进行详尽规划。
- **原子提交**：在上下文用尽前，确保所有重要进展已提交到 Git 或写入状态文件，避免信息丢失。
- **持续闭环**：每完成一个子任务（如 Phase 1.1），必须更新 `state_tracking` 系列文件并通知用户。
