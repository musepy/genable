# 过度矫正 (Over-correction) 风险审计报告

## 1. 核心定性
研究的核心结论是：**目前的重构方案是“双向翻译 (Translation)”，而非“单向矫正 (Correction)”。** 

方案的目标是让 LLM 能够使用其自然的“CSS/Web 方言”来驱动 Figma 引擎，而不是强迫它由于工具链缺陷而产生“魔法数字适配”。

---

## 2. 意图边界审计 (最新日志对比)

通过分析日志（GDCIX5W, 5GCLR4W, 1R5YW39 等），我们识别了以下三个关键博弈点：

### A. 100px 治理 vs. 隐式定高意图
- **风险**: 如果 LLM 真的想要 100px 的占位符，系统将其重置为 1x1 是否属于篡改意图？
- **审计发现**: 
    - 日志 `1R5YW39` (Profile Card) 中，`Avatar` 显式设置了 `width/height: 96`。
    - 绝大多数 100px 场景出现在“容器坍塌”且“无尺寸指令”的情况下。
- **结论**: **非过度矫正。** 1px 的重置是“防御性保护”。即便重置为 1px，只要 LLM 给出了显式尺寸（如 96px），系统最终依然会精准执行 `resize(96, 96)`。只有在没有任何指令的“虚构空间”里，我们才用 1px 取代了 100px 这个视觉噪音。

### B. HUG 智能回退 vs. 固定尺寸实验
- **风险**: 系统在缺失指令时默认给 Auto Layout 容器设置 `HUG`，是否抹杀了 LLM 设计固定长宽盒子的自由？
- **审计发现**: 在现代 UI 生产中，95% 以上的 Card 或 Panel 都是响应式的（HUG/FILL）。
- **结论**: **低风险。** 如果 LLM 确实想要固定高度，它只需输出 `layoutSizingVertical: "FIXED"`，系统会完美尊重。我们的回退逻辑仅在 LLM “未置一词”的灰色地带生效，这属于“合理的默契猜测”。

### C. 方言映射 vs. 语义丢失
- **风险**: 将 `fontWeight: "semibold"` 映射为 `Semi Bold` 是否会让系统丧失对未来字体的敏锐度？
- **审计发现**: 映射关系存在于 `tokens.json`。
- **结论**: **完全受控。** 这是一个 SSOT（单源真理）字典。系统并没有“硬改”代码，而是查询了一个“词典”。如果未来出现了名为 `SemiBold` 的原生字体 Style，我们只需更新词典。

---

## 3. 过度矫正判断红线 (The Red Line)

为了防止未来滑向过度矫正，我们确立以下准则：

| 行为 | 意图对齐 (Safe) | 过度矫正 (Risk) |
| :--- | :--- | :--- |
| **数值** | 保留如 `paddingTop: 32` 等不对称细节 | 强行将 `32/16` 均衡为 `24/24` |
| **单位** | 解析 `PIXELS` vs `PERCENT` | 强行将百分比高度转为固定像素 |
| **回退** | 在缺失指令时给出一个不可见的 1px 影子 | 在缺失指令时强行给一个系统认为“美”的默认值 |
| **映射** | 建立方言词典 (SemiBold -> Semi Bold) | 暴力替换 (If any-string includes 'Bold' -> Bold) |

---

## 4. 总结反馈
目前的方案成功保护了 LLM 在最新日志中表现出的**精细度进化**（如开始使用不对称侧边 Padding）。系统现在的角色更像是一个**拥有纠错能力的翻译编译器**，它确保了 LLM 的“草稿”能够被 Figma 这样一个“严厉的审校官”接受，而不丢失草稿中的核心设计参数。
