# 执行计划：推理层集成 (Reasoning Layer Implementation Plan)

> **目标**: 通过复用 UI Pro Max 的设计推理数据，为 Figma 生成以引入动态意图解析能力，彻底解决配置硬编码和维护难的问题。
> **策略**: 数据复用 (Data Reuse) + 适配器模式 (Adapter Pattern) + 客户端检索 (Client-side Search)。

## 核心架构决策

我们已确定采用 **"构建时转译 + 运行时检索"** 的架构方案。这一决策基于对软件工程原则（DRY, SSOT）的坚持以及对运行时环境限制的考量。

我们将不再手动维护孤立的 `constraints.json` 或硬编码的 Prompt 规则。相反，我们将建立一条自动化数据管道，将经过验证的 `ui-reasoning.csv`（以及其他关联 CSV）作为单一事实来源（SSOT）。这些数据将在构建阶段被转换为类型安全的 JSON 格式，并由 `minisearch` 驱动的轻量级检索引擎在运行时消费。

此方案能以最小的工程成本获得最大的智能提升，既避免了重复造轮子（移植 Python 逻辑），又规避了 Figma 插件环境的限制。

## 执行阶段分解

### 第一阶段：数据管道搭建 (The Data Pipeline)
**目标**: 建立从 CSV 到 TypeScript 的自动化桥梁。

我们将创建一个专门的构建脚本（adapter），负责读取 UI Pro Max 的原始 CSV 文件。该脚本不仅是简单的格式转换器，更是数据完整性的守门人。它将验证每一行数据的有效性，过滤掉不适用的字段，并将扁平的 CSV 结构重塑为嵌套的、语义化的 JSON 对象。重要的是，该脚本还将自动生成 TypeScript 类型定义（`.d.ts`），确保我们在后续开发中拥有完善的类型提示和编译时安全。

### 第二阶段：推理引擎集成 (The Reasoning Engine)
**目标**: 在 `DesignSystemLoader` 中引入智能检索能力。

我们将引入 `minisearch` 库作为核心检索组件。相比于手动移植 BM25 算法，使用这个成熟的库能提供更稳健的模糊搜索和前缀匹配能力。`DesignSystemLoader` 将演变为一个真正的“智能中枢”，它不再仅仅是根据 ID 加载静态文件，而是能够接收用户的自然语言意图（如“为金融科技初创公司设计一个仪表盘”），通过检索引擎查询最匹配的设计策略，并动态组装出相应的约束配置。

### 第三阶段：意图解析与注入 (Intent Resolution & Injection)
**目标**: 打通从用户输入到生成配置的全链路。

我们将定义一个清晰的 `DesignIntent` 接口，它承载了从对话中提取的语义信息（如类别、情绪、风格偏好）。在聊天模块中，简单的关键词匹配或轻量级的 LLM 提取将被用来填充这个意图对象。随后，这个意图对象被传递给 `DesignSystemLoader`，触发上一阶段构建的检索逻辑，最终生成一份“上下文感知”的 Prompt 和约束集，指导生成引擎产出高质量的设计。

## 信心与 ROI 分析

**高置信度 (High Confidence)**:
我们的信心来自于对底层数据的信任。`ui-reasoning.csv` 并非实验性数据，而是一套成熟的、包含上百条设计规则的知识库。通过复用它，我们实际上是站在了巨人的肩膀上。同时，`minisearch` 的引入消除了算法实现的风险，将复杂的检索问题简化为了标准的库调用。

**高投资回报率 (High ROI)**:
- **成本低**: 我们不需要重新编写成百上千条设计规则，也不需要调试复杂的 Python 到 JS 的移植代码。
- **收益大**: 系统将瞬间获得处理数十种不同设计场景的能力，且未来的规则更新只需在 CSV 中添加一行即可自动生效。维护成本被压缩到了最低，而系统的灵活性和智能度得到了质的飞跃。
