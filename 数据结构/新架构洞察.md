核心洞察：LLM 的能力被浪费了
LLM 写 HTML/React 很强，因为 HTML 就是一棵嵌套树，CSS 就是属性声明。LLM 在训练数据中见过几十亿个网页，对"怎么用嵌套结构描述 UI"这件事有极强的直觉。
但你的插件让 LLM 做了一件完全不同的事：一次一个节点地"打电话"给 Figma API。这就好比：LLM 明明会写整篇文章，但你让它一次只能说一个词，每说一个词都要等你确认。

二、为什么会这样：架构层面的分析
根本原因：把"生成"拆成了"对话"
当前架构是一个通用 Agent 循环：
while (没完成) {
  system_prompt + 所有历史消息 → 发给 LLM
  LLM 返回一个 tool call（createNode / batchOps / ...）
  执行 tool → 拿到结果
  把结果塞回历史
  继续下一轮
}
这个循环适合需要推理和决策的任务——比如"检查这个设计有没有对齐问题"、"帮我把这个组件改成深色模式"。因为 LLM 需要看到执行结果后决定下一步做什么。
但生成一个 Login 表单不需要这种推理。你脑子里已经知道 Login 表单长什么样：一个卡片，标题，两个输入框，一个按钮，一些链接。LLM 也知道。它不需要看到"标题创建成功"才能决定下一步是"创建输入框"。
这是一个"已知结构的批量输出"任务，却被当成了"需要逐步推理的对话"任务来处理。
类比
想象你让一个前端工程师写一个 Login 组件：

合理做法：他打开编辑器，写完整个 LoginForm.tsx，保存，完成
当前做法：他每写一行代码都要提交一次 PR，等 CI 跑完，看到"编译成功"后再写下一行

计算复杂度
为什么 token 消耗是 O(n²) 而不是 O(n)？
第 1 轮：LLM 处理 [system + prompt + node_1 结果] → ~4000 tokens
第 2 轮：LLM 处理 [system + prompt + node_1 + node_2 结果] → ~4500 tokens
第 3 轮：LLM 处理 [system + prompt + node_1 + node_2 + node_3 结果] → ~5000 tokens
...
第 n 轮：LLM 处理 [system + prompt + 所有前 n-1 个节点的结果] → ~(4000 + 500n) tokens
总消耗 ≈ 4000n + 500 × n(n-1)/2 = O(n²)
20 个节点 → 总处理量约 10 万 tokens。而如果一次性输出，只需要约 5000-8000 tokens。

三、LLM 实际在输出什么 vs 它擅长输出什么
当前：Figma 原子操作 JSON
每轮 LLM 输出这种东西：
json{
  "tool": "createNode",
  "params": {
    "type": "FRAME",
    "name": "Email Field Group",
    "parentId": "304:231",
    "layout": {
      "layoutMode": "VERTICAL",
      "gap": 8,
      "sizing": { "horizontal": "FILL", "vertical": "HUG" }

}
这不是 LLM 的最佳发挥形式。它需要记住 parentId: "304:231" 是哪个节点、当前 plan 进行到哪一步、哪些节点已经创建了。这些都是"工作记忆"负担，不是"设计能力"。
LLM 真正擅长的
LLM 在训练数据里见过几十亿个这种结构：
html<div class="flex flex-col gap-4 p-6 bg-white rounded-xl shadow-md max-w-md">
  <h2 class="text-2xl font-bold">Welcome Back</h2>

```

这 15 行代码完整描述了 Login 表单的结构、层级、样式、内容。LLM 一口气就能输出，不需要 26 轮对话。

## 四、你的场景分析：不同复杂度的设计

你提到了很好的分类，让我逐一分析：

### 小组件（按钮、输入框、badge）
- 嵌套 2-3 层，节点 3-8 个
- **当前方式**：5-10 轮迭代 → 可以接受但仍然浪费
- **理想方式**：1 次输出 → 节省 80% token

### 复合组件（登录表单、卡片、表格行）
- 嵌套 3-5 层，节点 10-30 个
- **当前方式**：20-40 轮迭代 → 明显太慢，容易出错（如日志所示）
- **理想方式**：1-2 次输出 → 核心节省区间

### 模块（hero section、pricing table、feature grid）
- 嵌套 4-6 层，节点 30-80 个
- **当前方式**：可能 50+ 轮，极可能触发 stale step 和 loop detection
- **理想方式**：分段输出（每段一个组件），3-5 次

### 完整页面（landing page，10-20 个模块）
- 节点 200+
- **当前方式**：不现实，context 窗口会爆
- **理想方式**：分层生成——先生成页面骨架（10-20 个模块容器），再逐模块填充

### 复杂矢量（icon、插图）
- 嵌套可能不深但 path data 很长，fill/stroke/effect 叠加
- **特殊性**：这不是结构问题，是数据量问题。SVG path 的 `d` 属性可能几千字符
- **理想方式**：icon 走 icon 库查找（已有 createIcon），插图走 SVG 导入而非 LLM 生成

### 关键观察

**这些场景的共同点是：LLM 在生成之前就知道完整结构。**

它不需要看到"Email Input 创建成功"才能决定"接下来创建 Password Input"。表单的结构是确定性的。LLM 的价值在于**从自然语言推断出正确的结构**，而不是在 26 轮对话中逐步构建。

## 五、关于 Skill / 前置知识注入的判断

### 你的 Skill 系统不是 RAG

先说清楚 RAG 是什么：
```
用户提问 → embedding 模型把问题变成向量 → 在向量数据库中搜索相似文档
→ 把找到的文档片段塞进 prompt → LLM 基于这些片段回答
```

你的系统是：
```
用户提问 → regex 匹配关键词（triggerPatterns）→ 注入对应 SKILL.md 文本
           → MiniSearch (BM25) 全文搜索知识库 → 返回匹配条目
区别：

RAG 用向量相似度（语义层面），你用关键词匹配（词汇层面）
RAG 的检索是自动的（LLM 不知道），你的检索是 tool call（LLM 主动调用 searchDesignKnowledge）
RAG 需要 embedding 模型，你用 MiniSearch 不需要

RAG 在这里不是更好的方案
你的直觉完全正确。原因：

知识库太小：几十个组件 anatomy + 少量 guidelines。BM25 全文搜索在小数据集上比向量搜索更精确——因为向量搜索需要足够多的数据才能学到有意义的语义空间
查询太结构化："button anatomy"、"card layout" 这种查询，关键词精确匹配比"语义接近"更可靠。向量搜索可能把 "button" 和 "switch toggle" 当成相似的，但你要的就是 button
真正的瓶颈不在知识检索：从日志看，knowledge 相关 tool 在这次生成中根本没被调用。LLM 直接跳过了知识查询，直接开始 createNode。这说明前置知识注入对生成质量的影响远不如生成策略本身
RAG 引入的复杂度不值得：在 Figma 插件沙箱里跑 embedding 模型需要额外的计算资源和延迟。换来的收益在小知识库场景下几乎为零

但 Skill 系统有优化空间
当前 skill 注入的问题不是"检索方法不对"，而是注入时机和利用率：

design-knowledge 是 dynamic 注入，但 triggerPattern 里没有 "login"、"form" 这种常见词 → 大多数生成请求不会激活它
project-ui-context 的 triggerPattern 有 "input"、"button"，但 LLM 在实际生成中跳过了 getProjectUIContext 和 getDesignSystemTokens 的调用 → skill 被注入了但没被使用
结果：生成的 Login 表单没有使用项目的 design tokens（颜色、字体、间距都是 LLM 自己编的）

七、改进方向：利用 LLM 的长处
核心思路：让 LLM 输出它最擅长的格式，由中间层翻译成 Figma 操作
LLM 擅长输出 HTML/JSX-like 嵌套结构。与其让它学一套 Figma 私有 DSL，不如：
P0：引入"组件树"生成模式
两阶段生成（利用 LLM 的代码生成能力）：
阶段 1 — 结构生成：LLM 一次性输出组件树（类 JSX 的 JSON）
json{
  "type": "FRAME", "name": "Login Card",
  "layout": "VERTICAL", "padding": 32, "gap": 24,
  "fills": ["#FFFFFF"], "cornerRadius": 16,
  "width": 400, "sizing": "HUG",
  "children": [
    { "type": "TEXT", "name": "Title", "text": "Welcome Back",
      "fontSize": 28, "fontWeight": "Bold" },
    { "type": "TEXT", "name": "Subtitle", "text": "Enter your credentials",
      "fontSize": 14, "color": "#6B7280" },
    { "type": "FRAME", "name": "Email Field", "layout": "VERTICAL", "gap": 6,
      "children": [
        { "type": "TEXT", "text": "Email", "fontSize": 14, "fontWeight": "Medium" },
        { "type": "FRAME", "name": "Input", "layout": "HORIZONTAL",
          "padding": [12, 16], "cornerRadius": 8, "stroke": "#D1D5DB",
          "children": [
            { "type": "TEXT", "text": "you@email.com", "color": "#9CA3AF" }
          ]}
      ]},
    ...整个树一次输出
  ]
}
这个 JSON 结构跟 HTML 嵌套几乎一一对应——LLM 不需要学新东西。
阶段 2 — 翻译渲染：中间层（你已有的 Normalizer + RenderOrchestrator）把这棵树转换成 Figma API 调用。
关键：你的 batchOperations 已经支持嵌套 children，createNode 已经支持 inline layout 和 styles。基础设施大部分已经有了。核心工作是：

新增一个 generateComponentTree tool，接受完整的嵌套 JSON
在 RenderOrchestrator 中递归处理这棵树
在 skill prompt 中引导 LLM 使用这个工具而不是逐个 createNode

关于嵌套深度的问题
你之前担心 LLM 输出深层嵌套 JSON 会"遵循能力下降"。这个担心合理，但需要分情况：
小/中组件（3-5 层嵌套）：LLM 完全 hold 得住。它训练时见过无数 3-5 层嵌套的 HTML。
大页面（10+ 模块，总嵌套 8+ 层）：确实不应该一次输出。解法是分层递归：

第 1 轮：输出页面骨架（只有模块级容器，不含子节点）
第 2-N 轮：逐模块填充子节点

这和前端工程师的工作方式一样：先搭页面布局骨架，再一个个填充 section 的内容。
复杂矢量/icon：不应该让 LLM 生成 SVG path。走 icon 库（你已有 createIcon）或 SVG 资源导入。
P1：强制在 planning 阶段完成知识查询
当前问题：LLM 被注入了 design-knowledge 和 project-ui-context skill，但在 EXECUTION 阶段跳过了查询。
改法：在 PLANNING 模式的 prompt 中明确要求"必须先调用 getDesignSystemTokens 和 getProjectUIContext，拿到 tokens 后再开始生成"。把知识查询从"可选"变成"必选前置步骤"。
P2：AgentRuntime 解耦
1914 行的 agentRuntime.ts 承担了太多职责。建议拆分：

ContextManager：消息历史管理、token 预算、压缩
LoopDetector：语义签名、重复检测、恢复
PlanExecutor：计划推进、stale step 检测
AgentRuntime：只保留主循环和 mode 切换

这不直接改善生成质量，但让后续优化更容易实施。
